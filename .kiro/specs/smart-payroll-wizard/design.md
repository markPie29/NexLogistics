# Design Document: Smart Payroll Wizard

## Overview

This design upgrades the existing payroll run wizard at `app/(app)/payroll/run/page.tsx` into a smarter, Philippine-BIR-compliant payroll generation experience. The current implementation already has the 5-step structure (Period → Eligible Trips → Compute Earnings → Adjustments → Review & Generate) and uses the existing computation engine (`buildDriverSummary`, `computeGovernmentDeductions`, `getEligibleTripsForDriver`). The redesign adds:

1. **Smart period detection** — auto-detect next cutoff from closed period history
2. **Pending-trip blocking** — configurable threshold that gates advancement
3. **Enhanced warnings system** — categorized errors/warnings/info with blocking semantics
4. **Duplicate detection** — cross-period overlap checking at review time
5. **Audit trail** — record who generated, when, and with what parameters
6. **Polish** — helper text, "Step X of 5" footer, compliance badge, state persistence

**Design philosophy**: Surgical upgrade of the existing page. Reuse the existing computation engine, stores, and types. Extract pure logic into utility functions for testability. No new stores or routes — only refinements to the wizard page and a new utility module.

**Key architectural decisions:**

1. **Extract pure logic to `lib/payroll/wizard-utils.ts`** — Period detection, name formatting, pay date computation, overlap detection, warning generation, employee filtering, and adjustment impact calculation become testable pure functions.
2. **Keep wizard page as orchestrator** — The page component calls utility functions and manages step state via React `useState`. No new Zustand store for wizard state (the existing approach of component-level state is sufficient since the wizard is a single page).
3. **Reuse existing computation engine verbatim** — `buildDriverSummary`, `computeGovernmentDeductions`, `computeTripPayroll`, and `getEligibleTripsForDriver` remain untouched.
4. **Audit trail stored on PayrollPeriod** — Add `auditTrail` field to the period record (generated by/at already exists; extend with employee count and grand total).

## Components and Interfaces

### New Module: `lib/payroll/wizard-utils.ts`

```typescript
// ─── Period Detection ───

interface PeriodConfig {
  name: string;
  startDate: string;   // ISO date (YYYY-MM-DD)
  endDate: string;
  payDate: string;
  cutoff: "A" | "B";
}

/**
 * Detect next payroll period from current date.
 * Cutoff A = 1st-15th, Cutoff B = 16th-last day of month.
 */
function detectPeriodFromDate(date: Date): PeriodConfig;

/**
 * Detect next sequential period based on the most recently closed/paid period.
 * If last closed is Cutoff A of June → next is Cutoff B of June.
 * If last closed is Cutoff B of June → next is Cutoff A of July.
 */
function detectNextPeriodFromHistory(
  periods: PayrollPeriod[]
): PeriodConfig;

/**
 * Format period name: "{Month} {startDay}-{endDay}, {year} · Cut-off {A|B}"
 */
function formatPeriodName(startDate: string, endDate: string): string;

/**
 * Compute pay date: endDate + 5 calendar days.
 */
function computePayDate(endDate: string): string;

// ─── Overlap Detection ───

interface OverlapResult {
  hasOverlap: boolean;
  overlappingPeriod?: PayrollPeriod;
  isExactMatch: boolean;
  isBlocking: boolean;   // true if overlapping period is "paid" or "closed"
}

/**
 * Check if candidate period overlaps any existing period.
 */
function detectPeriodOverlap(
  startDate: string,
  endDate: string,
  existingPeriods: PayrollPeriod[]
): OverlapResult;

// ─── Employee Eligibility ───

interface EmployeeEligibility {
  employeeId: string;
  employeeName: string;
  employeeType: "driver" | "helper" | "office";
  approvedTrips: number;
  pendingTrips: number;
  rejectedTrips: number;
  totalTrips: number;
  payrollMode: PayrollMode | null;
  isFixedSalary: boolean;
}

/**
 * Filter and aggregate employee eligibility for the period.
 * Returns only employees who will receive pay (approved trips > 0 OR fixed/hybrid profile).
 */
function filterEligibleEmployees(
  drivers: Driver[],
  helpers: Helper[],
  profiles: DriverPayrollProfile[],
  trips: Trip[],
  startDate: string,
  endDate: string
): EmployeeEligibility[];

// ─── Blocking Threshold ───

/**
 * Determine if pending trip count exceeds threshold.
 */
function isBlockedByPendingTrips(
  pendingCount: number,
  threshold: number
): boolean;

// ─── Adjustment Impact ───

interface AdjustmentImpact {
  totalIncentives: number;
  totalDeductions: number;
  netImpact: number;   // incentives - deductions
  itemCount: number;
}

/**
 * Compute net impact of unlocked adjustments within period range.
 */
function computeAdjustmentImpact(
  incentives: Incentive[],
  deductions: Deduction[],
  startDate: string,
  endDate: string
): AdjustmentImpact;

// ─── Smart Warnings ───

type WarningSeverity = "error" | "warning" | "info";

interface PayrollWarning {
  severity: WarningSeverity;
  message: string;
  employeeId?: string;
}

/**
 * Generate categorized warnings from computed results.
 * Sorted: errors first, then warnings, then info.
 */
function generateSmartWarnings(
  computed: { driverId: string; driverName: string; summary: PayrollSummary }[],
  allTrips: Trip[],
  profiles: DriverPayrollProfile[],
  drivers: Driver[]
): PayrollWarning[];

// ─── Cross-Period Duplicate Detection ───

interface DuplicateFlag {
  employeeId: string;
  employeeName: string;
  conflictingPeriodName: string;
}

/**
 * Detect employees who appear in existing periods with overlapping dates.
 */
function detectCrossPeriodDuplicates(
  currentStartDate: string,
  currentEndDate: string,
  computedEmployeeIds: string[],
  existingPeriods: PayrollPeriod[],
  existingSummaries: PayrollSummary[],
  employeeNameMap: Record<string, string>
): DuplicateFlag[];

// ─── Grand Total ───

/**
 * Sum net pay across all categories.
 */
function computeGrandTotal(
  driverSummaries: PayrollSummary[],
  helperSummaries: PayrollSummary[],
  officeResults: { netPay: number }[],
  partnerResults: { totalPayout: number }[]
): number;
```

### Modified Component: `app/(app)/payroll/run/page.tsx`

Changes to the existing wizard page:

1. **Import `wizard-utils.ts`** and replace inline logic with utility calls
2. **Add `blockingThreshold` state** (default: 5) with inline settings control in Step 2
3. **Add `overlapWarning` state** computed when period dates change (Step 1)
4. **Add reminder banner in Step 4** (already partially exists — enhance with running total)
5. **Add cross-period duplicate detection in Step 5** using `detectCrossPeriodDuplicates`
6. **Add audit trail fields** to the generated PayrollPeriod record
7. **Add "Step X of 5" text** in footer (already exists — keep as-is)
8. **Enhance stepper styling** with brand-navy/emerald-green/gray per spec
9. **Add employee selector for helpers/office in adjustment dialogs** (currently only drivers)

### No New Stores Required

The wizard state (period config, computed results, threshold) lives in React component state as it does today. This is appropriate because:
- The wizard is a single page — no cross-page navigation needed
- Leaving the page discards the run (Cancel behavior)
- localStorage persistence would be confusing (stale wizard data on revisit)

## Data Models

### Extended PayrollPeriod (audit trail fields)

The existing `PayrollPeriod` interface already has `generatedBy` and `generatedAt`. We extend the generated record with additional metadata:

```typescript
// Added to the period record at generation time (stored in existing fields):
{
  generatedBy: string;       // ← already exists
  generatedAt: string;       // ← already exists
  notes: string;             // ← use for audit: "12 drivers, 4 helpers, 3 office, 2 partners. Grand total: ₱XXX"
}
```

No schema changes needed — the existing `notes` field on `PayrollPeriod` accommodates the audit summary. The `generatedBy`/`generatedAt` fields already track who and when.

### Wizard Step Validation Rules

| Step | Validation Rule | Blocks Next? |
|------|----------------|--------------|
| 1 - Period | Valid name, start ≤ end, no exact overlap with paid/closed | Yes |
| 2 - Trips | Pending trips ≤ threshold, at least 1 qualifying employee | Yes |
| 3 - Compute | Computation performed, no duplicate employee IDs | Yes |
| 4 - Adjustments | None (informational step) | No |
| 5 - Review | Computation results exist | Yes (disables Generate) |

### Warning Generation Rules

| Condition | Severity | Blocks? |
|-----------|----------|---------|
| Net pay ≤ 0 | error | Yes (blocks Step 3→4) |
| Duplicate employee ID in results | error | Yes |
| Zero gross pay with active profile | warning | No |
| Driver with trips but no profile | info | No |
| Withholding tax computed | info | No |
| Cross-period employee overlap | warning (Step 5) | No |

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Period Cutoff Detection

*For any* date, `detectPeriodFromDate` SHALL return Cutoff A with startDate on the 1st and endDate on the 15th if the date is between the 1st and 15th (inclusive), OR Cutoff B with startDate on the 16th and endDate on the last day of the month if the date is between the 16th and the last day.

**Validates: Requirements 1.1**

### Property 2: Period Name and Pay Date Derivation

*For any* valid start date and end date where start ≤ end, `formatPeriodName` SHALL produce a string matching the pattern `"{Month} {startDay}-{endDay}, {year} · Cut-off {A|B}"` AND `computePayDate` SHALL return a date exactly 5 calendar days after the end date.

**Validates: Requirements 1.2, 1.3**

### Property 3: Next Sequential Period Detection

*For any* non-empty array of closed/paid periods, `detectNextPeriodFromHistory` SHALL return the period immediately following the most recently ended period: if the last period is Cutoff A (ends on 15th), the next SHALL be Cutoff B of the same month; if the last period is Cutoff B (ends on last day), the next SHALL be Cutoff A of the following month.

**Validates: Requirements 1.4**

### Property 4: Period Overlap Detection

*For any* candidate date range and array of existing periods, `detectPeriodOverlap` SHALL return `hasOverlap: true` if and only if there exists an existing period whose date range intersects with the candidate range, AND `isBlocking: true` if and only if that overlapping period has status "paid" or "closed".

**Validates: Requirements 1.7**

### Property 5: Employee Eligibility Filtering

*For any* set of employees (drivers + helpers), payroll profiles, and trips within a period, `filterEligibleEmployees` SHALL return exactly those employees who have at least one approved trip in the period OR who have a payroll profile with mode "fixed_salary" or "fixed_plus_trip" or "hybrid", and SHALL exclude all others.

**Validates: Requirements 2.1, 2.9**

### Property 6: Trip Count Aggregation

*For any* employee and set of trips within a period, the aggregated counts SHALL satisfy: `approvedTrips + pendingTrips + rejectedTrips ≤ totalTrips` AND each count SHALL equal the number of trips with the corresponding `approvalStatus` value for that employee within the date range.

**Validates: Requirements 2.2**

### Property 7: Blocking Threshold Enforcement

*For any* non-negative pending trip count and positive threshold value, `isBlockedByPendingTrips` SHALL return `true` if and only if `pendingCount > threshold`.

**Validates: Requirements 2.6**

### Property 8: Earnings Computation Consistency

*For any* driver with a valid payroll profile, set of approved trips, and trip rates, `buildDriverSummary` SHALL produce a summary where: `grossPay = baseSalary + tripEarnings + incentives + allowances + overtimeAmount` AND `netPay = grossPay - totalDeductions` AND `totalDeductions = sss + philhealth + pagibig + tax + cashAdvance + otherDeductions`. For "percentage" mode employees, `commissionAmount` SHALL equal `commissionPercent / 100 × trip.fare` for each trip.

**Validates: Requirements 3.1, 3.7**

### Property 9: Statutory Deduction Caps and Tax Brackets

*For any* gross pay value ≥ 0 and any profile configuration, `computeGovernmentDeductions` SHALL produce: SSS ≤ ₱1,350 AND PhilHealth ≤ ₱2,500 AND Pag-IBIG ≤ ₱200 (and Pag-IBIG = 0 when gross ≤ ₱1,500). For withholding tax, the computed tax SHALL match the BIR TRAIN Law 2026 monthly bracket table applied to (grossPay - SSS - PhilHealth - Pag-IBIG).

**Validates: Requirements 3.2**

### Property 10: Grand Total Summation

*For any* set of computed results across all employee categories, `computeGrandTotal` SHALL return a value equal to the sum of all individual net pay amounts (drivers + helpers + office staff) plus all partner payout amounts.

**Validates: Requirements 3.6**

### Property 11: Adjustment Filtering and Net Impact

*For any* set of incentives and deductions, `computeAdjustmentImpact` SHALL include only items where `payrollPeriodId` is null (unlocked) AND `createdAt` falls within [startDate, endDate], AND the `netImpact` SHALL equal sum(included incentive amounts) minus sum(included deduction amounts).

**Validates: Requirements 4.4, 4.5**

### Property 12: Cross-Period Duplicate Detection

*For any* set of computed employee IDs, current period dates, and existing period records with summaries, `detectCrossPeriodDuplicates` SHALL flag an employee if and only if that employee's ID appears in a summary belonging to an existing period whose date range overlaps with the current period's date range.

**Validates: Requirements 5.3**

### Property 13: Step Validation Gating

*For any* wizard state at step N, clicking "Next" SHALL advance to step N+1 only if the step-specific validation passes: Step 1 requires non-empty name AND startDate ≤ endDate AND no blocking overlap; Step 2 requires pendingTrips ≤ threshold AND at least one eligible employee; Step 3 requires computation performed AND no duplicate entries.

**Validates: Requirements 6.4**

### Property 14: Wizard State Persistence

*For any* data entered in the wizard (period config, computed results, adjustments), navigating forward to step N+1 and then back to step N SHALL preserve all previously entered data without loss or reset.

**Validates: Requirements 6.5**

### Property 15: Smart Warning Generation and Categorization

*For any* set of computed payroll results, `generateSmartWarnings` SHALL: produce an error for each employee with netPay ≤ 0, produce a warning for each employee with grossPay = 0 despite an active profile, produce an info notice for each driver with trips but no profile, produce an info notice when any tax deduction > 0, AND the returned array SHALL be sorted with errors first, then warnings, then info.

**Validates: Requirements 7.1**

## Error Handling

| Scenario | Handling |
|----------|----------|
| No closed periods in store (first-ever run) | `detectNextPeriodFromHistory` falls back to `detectPeriodFromDate(new Date())` |
| Store read error (localStorage corruption) | Zustand persist middleware handles this — stores reset to seed data. Wizard shows computed=[] state. |
| Zero qualifying employees | Step 2 shows empty state, Next disabled with message |
| Net pay negative (over-deducted) | Error warning surfaced, blocks Step 3→4 progression |
| Duplicate employee in computed results | Error warning, blocks Step 3→4 |
| Payroll generation fails (addPeriod throws) | Catch in generate(), show error toast via sonner, remain on Step 5 |
| Period overlap with closed period | Blocking overlap — Step 1 Next disabled with inline warning |
| Period overlap with draft/computing period | Non-blocking warning — show duplicate indicator but allow proceed |

## Testing Strategy

### Property-Based Tests (via `fast-check`)

The extracted pure functions in `lib/payroll/wizard-utils.ts` and the existing computation engine functions are ideal candidates for property-based testing:

- **Target**: `detectPeriodFromDate`, `formatPeriodName`, `computePayDate`, `detectNextPeriodFromHistory`, `detectPeriodOverlap`, `filterEligibleEmployees`, `isBlockedByPendingTrips`, `computeGovernmentDeductions`, `computeGrandTotal`, `computeAdjustmentImpact`, `detectCrossPeriodDuplicates`, `generateSmartWarnings`
- **Library**: `fast-check` (already available in the JS/TS ecosystem, lightweight)
- **Iterations**: Minimum 100 per property
- **Tag format**: `Feature: smart-payroll-wizard, Property {N}: {title}`

Each correctness property (1–15) maps to one property-based test that generates random valid inputs and asserts the universal property holds.

### Unit Tests (via Vitest)

Example-based tests for:
- UI rendering: stepper colors at each step, helper text presence, BIR badge content
- UI interactions: inline approve, dialog open/close, cancel navigation
- Edge cases: empty employee list, zero-trip period, first-ever payroll run
- Integration: adding incentive persists to store, generate creates period record

### Test File Structure

```
__tests__/
  lib/payroll/
    wizard-utils.property.test.ts   ← Property-based tests (Properties 1-12, 15)
    wizard-utils.unit.test.ts       ← Example-based edge cases
  app/payroll/
    run-wizard.test.tsx             ← Component tests (Properties 13-14, UI examples)
```

### What Is NOT Tested via PBT

- UI rendering/layout (snapshot or visual regression)
- Store persistence to localStorage (integration)
- Router navigation (mock-based unit test)
- Toast notifications (mock-based unit test)
